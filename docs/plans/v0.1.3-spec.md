## 1. Context

### 1.1 The Bug Surface — Two Distinct Shapes

**Critical clarification (post-spec-review, advisor caught conflation):** F25 and F26 reference **two different YAML key-shapes**. Both bite skills, but the fixes differ. Treating them as one is a spec-bug.

#### Shape (β) — F26 Pathological (Apple-Notes-Vintage)

Apple Notes / Obsidian Importer (early-vintage) imports leave YAML frontmatter with quoted keys that **contain a trailing colon INSIDE the quotes**, plus the YAML key-value colon outside:

```yaml
---
"created:": 2024-03-14
"modified:": 2026-04-13
"description:": Original Apple Notes export
tags:
  - AppleNoteImport
---
```

The valid YAML interpretation: a key literally named `created:` (with the colon as part of the key string). Strict YAML parsers accept this — but **no human author intends this**. It is a serialization artifact. **F26 = invalid-as-author-intended.**

#### Shape (α) — F25 Standard Quoted-Key

Standard YAML allows quoting keys for various reasons (escaping special chars, version-specific tooling, hand-typed style). The form `"<key>":` (closing `"` then `:`, **no colon inside quotes**) is **valid YAML**:

```yaml
---
created: 2024-03-14
"description": A standard quoted-key value
tags:
  - SomeTag
---
```

YAML parsers correctly parse this as `description` → `A standard quoted-key value`. The skill bug is that today's Step 2 filter regex `^([A-Za-z_][A-Za-z0-9_-]*)\s*:` only matches plain identifiers — quoted-key form (which starts with `"`) is missed, even though it is valid YAML. **F25 = filter regex too narrow.**

#### Why distinguishing matters

| Aspect | F25 (shape α — standard) | F26 (shape β — pathological) |
|--------|--------------------------|------------------------------|
| YAML validity | Valid | Invalid-as-author-intended (importer artifact) |
| Detection regex | `^\s*"[^"]+"\s*:` (no inside-colon) | `^\s*"[^"]+:"\s*:` (inside-colon required) |
| Fix policy | Filter-regex broadening (read both shapes) | Repair via recipe (f) normalize |
| Sanity-check verdict | `OK_QUOTED` (proceed normally) | `BROKEN_KEYS_INSIDE_COLON` (Class-C) |
| User-facing impact | None after fix — file processes normally | Same — but file shape changes (key un-quoted) |

**Empirical anchor (heutiger Report L71):** F25 file 2 (LinkedIn GPT) was `"description":` (standard form, shape α). The Pre-Run-Memo Stop caught it before write because filter regex missed the quoted form, marked the file as eligible despite already having a description. Fix: broaden filter regex.

**Empirical anchor (heutiger Report L72):** F26 cluster 60 files have `"<key>:":` shape (inside-colon, shape β). property-enrich Cell 1 missed them because its hardcoded normalize-list (only `"created:"` and `"modified:"`) doesn't generalize. Fix: recipe (f) generalized normalize.

### 2.1 F26-Cluster — Inside-Colon Quoted-Key Normalization (cross-skill)

> **Note:** F26 specifically targets shape β (`"<key>:":` with inside-colon). Standard quoted-keys (shape α) are valid YAML and are handled by F25 fix in § 2.2 — NOT by recipe (f).

**Where:** `property-enrich` (new Step 2a), `note-rename` (Step 4a generalize), `inbox-sort` (Step 5a generalize).

**Today (note-rename + inbox-sort):**
> "If the YAML contains `"created:"` or `"modified:"` (quoted with embedded colon), normalize to `created` / `modified` and persist immediately."

Hardcoded key-list. Misses `"description:"`, `"tags:"`, `"title:"`, `"aliases:"`, `"status:"`, `"type:"`, etc. — all shape β.

**v0.1.3 spec — generalized for shape β only:**
> "If any frontmatter line matches the inside-colon quoted-key pattern `^\s*"[^"]+:"\s*:` (any key wrapped in double-quotes WITH trailing colon inside quotes), normalize the key by stripping the surrounding `"` and trailing-`:` from inside the quotes. Persist immediately. Repeat scan until no inside-colon quoted-keys remain (idempotent fixpoint)."

**Per-line regex pattern (safe, single-line input — F26-specific):**

```python
F26_INSIDE_COLON_PATTERN = re.compile(r'^(\s*)"([^"]+):"\s*:(.*)$')
```

The crucial part: `:"` BEFORE the closing quote AND `\s*:` AFTER. Both colons required. This is what makes F26 distinct from F25 standard quoted-keys.

**Match groups:**
1. Leading whitespace (preserve indentation)
2. Key name (without surrounding quotes, without trailing-colon-inside-quotes)
3. Everything after the value-colon (the value, possibly with trailing comment)

**Replacement line:** `f"{group1}{group2}:{group3}"` — exact value preserved, only key shape changed (un-quoted, inside-colon stripped).

**Duplicate-key resolution (post-normalization):**

After normalization, two lines may now claim the same key (e.g. broken `"created:": 2024-03-14` + plain `created: 2025-01-01` both become `created:`). Policy:

| Case | Action |
|------|--------|
| Both inside-colon-form and plain-key existed before normalization | Keep first occurrence (= the inside-colon-original, normalized) — assume it has the original/correct value. Remove subsequent occurrences. Log as Class-D finding. |
| Two inside-colon forms collide (same key name) | Keep first. Remove second. Log as Class-D finding. |
| No collision (most cases) | Idempotent normalize, no further action. |

**Idempotency check:** Re-running on already-normalized frontmatter is a no-op (no line matches the F26 pattern).

**Does NOT touch shape α:** Standard quoted-keys like `"description":` (no inside-colon) are valid YAML and pass through unchanged. F25 fix in § 2.2 handles those.

### 2.2 F25 — `property-describe` Filter-Regex Permissiveness

**Where:** `property-describe` Step 2 (Filter — identify notes needing descriptions).

**Today's regex:**

```python
# F25 BUG — too narrow
DESC_KEY_PATTERN = re.compile(r'^([A-Za-z_][A-Za-z0-9_-]*)\s*:')
```

This matches plain identifier keys only. Quoted-key form `"description":` (shape α — valid YAML) does NOT start with `[A-Za-z_]` (starts with `"`), regex fails → key treated as missing → file flagged eligible → would write duplicate `description:` line. Live-reproduction in GR-2 Cell (4) file 2 (LinkedIn GPT, byte-confirmed `"description":` shape per heutiger Report L71).

**v0.1.3 fix — broaden the regex to accept BOTH shapes:**

```python
# F25 fix — accept plain-key OR standard-quoted-key (shape α)
# Does NOT accept inside-colon shape β — that goes through recipe (f) normalize first
DESC_KEY_PATTERN = re.compile(r'''
    ^
    (?:
        ([A-Za-z_][A-Za-z0-9_-]*)        # plain identifier
        |
        "([^":]+)"                        # standard quoted-key (no inside-colon!)
    )
    \s*:
''', re.VERBOSE)
```

**Critical:** the inner `[^":]+` (NO inside-colon allowed) is what distinguishes F25 (standard quoted-key) from F26 (inside-colon). The F26 form `"description:"` starts with `"description:` — the `:` inside the bracket forbids the match. F26 must be normalized first via § 2.1 recipe (f).

**Defense-in-depth via two-stage check (Step 2 Filter):**

```python
# Stage A: pre-flight sanity-check (per references/yaml-sanity.md)
verdict = sanity_check(frontmatter_lines)
if verdict == "BROKEN_KEYS_INSIDE_COLON":
    SKIP file with Class-C finding "broken-yaml: F26 inside-colon shape detected — run property-enrich first"
    continue
if verdict in ("MULTIPLE_FRONTMATTER_BLOCKS", "UNCLOSED_FRONTMATTER"):
    SKIP file with Class-A finding
    continue
# verdict in ("OK", "OK_QUOTED", "OK_NO_FRONTMATTER") → proceed

# Stage B: existing-description detection (uses broadened regex)
existing_description = find_description_value(frontmatter_lines)  # uses DESC_KEY_PATTERN
if existing_description and len(existing_description) >= 10:
    SKIP file (already has description)
```

**Why this split:**
- F25 (shape α) is valid YAML — broaden the regex, process the file normally. No skip, no repair.
- F26 (shape β) is broken-as-author-intended YAML — boundary-respect skip in property-describe (additive-only contract). User runs enrich first to repair, then re-runs describe.

**User flow under F26:** describe skips with finding "run property-enrich first" → enrich's Step 2a normalizes (recipe f) → user re-runs describe → file now has plain-key `description: ...` (or quoted-without-inside-colon if the original used shape α form) → broadened regex catches it → skip-as-eligible.

### 2.4 Pre-Write YAML Sanity Check (Defense-in-Depth)

**New reference doc:** `references/yaml-sanity.md` (full draft in § 5.1 below).

**Where called:** Step-zero of any property-write workflow. ALL 4 launch-scope skills (and future `property-classify`) call it before recipe (b/c/d/e) writes.

**Detection patterns (initial v0.1.3 — extensible later):**

| Pattern | Severity | Verdict | Repair-Policy | Skip-Policy |
|---------|----------|---------|--------------|-------------|
| **Inside-colon quoted-key** `^\s*"[^"]+:"\s*:` (shape β — F26) | Class-C broken-key | `BROKEN_KEYS_INSIDE_COLON` | repair via § 2.1 recipe (f) | skip-with-finding (read-only-skills) |
| **Standard quoted-key** `^\s*"[^":]+"\s*:` (shape α — F25) | n/a (valid YAML) | `OK_QUOTED` | n/a — no repair needed | n/a — proceed; skill regexes must accept both shapes |
| Multiple frontmatter blocks (≥2 `---/---` pairs) | Class-A corruption | `MULTIPLE_FRONTMATTER_BLOCKS` | n/a — no repair | skip-with-finding (all skills, route to user-review per note-rename's existing corrupted-file-detection) |
| No frontmatter close (`---` open without close) | Class-A corruption | `UNCLOSED_FRONTMATTER` | n/a — no repair | skip-with-finding (all skills) |
| Trailing whitespace-only frontmatter (whitespace between final field and close `---`) | Class-D cosmetic | logged as finding only | optional repair | proceed (non-blocking) |

**Critical detection-precedence rule:**

Walk inside-colon pattern FIRST, standard quoted-key SECOND. The inside-colon pattern is a strict subset (must contain inside-colon); standard pattern matches any quoted-key WITHOUT inside-colon. A line like `"description:":` matches inside-colon (verdict β), NOT standard (because the `:` inside `[^":]+` is forbidden). A line like `"description":` matches standard (verdict α), NOT inside-colon.

**Per-skill policy decision:**

| Skill | On Class-A | On `BROKEN_KEYS_INSIDE_COLON` (shape β) | On `OK_QUOTED` (shape α) | On Class-D cosmetic |
|-------|-----------|---------------------------------|----------------------------------|---------------------|
| `property-enrich` | skip + finding (route to user) | repair via Step 2a recipe (f) | proceed; skill regex matches both shapes | proceed |
| `note-rename` | use existing corrupted-file-detection | repair via Step 4a recipe (f) | proceed; skill regex matches both shapes | proceed |
| `inbox-sort` | skip + finding (route to user) | repair via Step 5a recipe (f) | proceed; skill regex matches both shapes | proceed |
| `property-describe` | skip + finding | skip + Class-C finding (NOT repair — boundaries; user runs enrich first) | proceed; broadened regex catches both shapes | proceed |
| `property-classify` (v0.2.0) | skip + finding | skip + finding | proceed; broadened regex | proceed |

Defense-in-depth lives in the **sanity-check call**: skills that already have repair logic in their workflow (enrich, rename, sort) are calling sanity-check as a Step-zero pre-flight. If sanity-check returns Class-A, skill skips. If Class-C, skill calls its own repair step. If both succeed, skill proceeds normally.

### 3.2 F2 — German Date-Format Normalization

**Where:** `property-enrich` Source Hierarchy.

**Today:** Hierarchy tries ISO formats only. German `30.01.2026, 17:02:59` (DACH-region critical) is not parseable → Source-Hierarchy falls through to filesystem birthtime even though a valid date is in YAML.

**v0.1.3 fix:** Add normalization step BEFORE the Hierarchy parser walks ISO formats:

```python
GERMAN_DATE_PATTERN = re.compile(r'^(\d{1,2})\.(\d{1,2})\.(\d{4})(?:,\s*(\d{1,2}):(\d{2})(?::(\d{2}))?)?$')

def normalize_german_date(value: str) -> str | None:
    m = GERMAN_DATE_PATTERN.match(value.strip())
    if not m:
        return None
    day, month, year = m.group(1), m.group(2), m.group(3)
    hour, minute, second = m.group(4), m.group(5), m.group(6)
    iso = f"{year}-{int(month):02d}-{int(day):02d}"
    if hour:
        iso += f"T{int(hour):02d}:{minute}:{second or '00'}"
    return iso
```

Called as first transformation when reading YAML `created` value. If match → use normalized ISO. If no match → fall through to existing parser logic.

## 5. New Reference Documents

### 5.1 `references/yaml-sanity.md` (full draft)

```markdown
# YAML Sanity — Pre-Write Defense-in-Depth

## Purpose

Every property-writing skill calls this sanity-check BEFORE attempting any
YAML edit (recipes b/c/d/e per `references/yaml-edits.md`). The check returns
a verdict the skill uses to decide: proceed, repair, or skip-with-finding.

## When to call

Step zero of any property-write workflow:
- `property-enrich` Step 2a (before Compute / Source Hierarchy)
- `property-describe` Step 2 (before Filter — eligibility check)
- `note-rename` Step 4a (before Classify cooldown logic)
- `inbox-sort` Step 5a (before Cooldown evaluation)
- `property-classify` (v0.2.0+) Step 2 (before Type/Status detection)

## Procedure

1. Read frontmatter lines per `references/yaml-edits.md` recipe (a).
2. If no frontmatter exists: return verdict `OK_NO_FRONTMATTER`.
3. Walk frontmatter lines. Match each line against detection patterns (§ 3
   below). Collect findings.
4. Return verdict + finding list.

## Verdicts

| Verdict | Meaning | Action by skill |
|---------|---------|-----------------|
| `OK` | Frontmatter is well-formed, only plain-identifier keys | Proceed normally |
| `OK_QUOTED` | Frontmatter has standard quoted-keys (shape α — `"key":`, valid YAML) but no inside-colon shape and no other issues | Proceed normally; skill regexes must accept both plain and standard-quoted forms |
| `OK_NO_FRONTMATTER` | File has no frontmatter to inspect | Proceed normally — recipe (c) creates one if needed |
| `BROKEN_KEYS_INSIDE_COLON` | One or more shape β inside-colon patterns detected (`"<key>:":`) — F26 | Repair via recipe (f) (if skill is repair-capable) OR skip + Class-C finding (additive-only skills) |
| `MULTIPLE_FRONTMATTER_BLOCKS` | Two or more `---/---` pairs at frontmatter-boundary positions | Skip + Class-A finding. Route to `note-rename` corrupted-file-detection if not already there. |
| `UNCLOSED_FRONTMATTER` | First line `---` but no matching closing `---` | Skip + Class-A finding. User-investigation required. |
| `INVALID_YAML` | Detection patterns OK but YAML still unparseable (rare) | Skip + Class-A finding. Manual review. |

**Verdict-priority:** A file may match multiple patterns. Verdict order: `MULTIPLE_FRONTMATTER_BLOCKS` > `UNCLOSED_FRONTMATTER` > `BROKEN_KEYS_INSIDE_COLON` > `OK_QUOTED` > `OK_NO_FRONTMATTER` > `OK`. Highest matching verdict wins. Cosmetic findings logged regardless of verdict.

## Detection patterns

### Pattern 1 — Inside-colon quoted-key (shape β — F26)

**Per-line regex (single-line input — safe per yaml-edits.md):**

```python
F26_INSIDE_COLON_PATTERN = re.compile(r'^(\s*)"([^"]+):"\s*:(.*)$')
```

The crucial structure: `:"` BEFORE the closing quote AND `\s*:` AFTER the closing quote. Both colons are required to match.

**Match:** indent + key-without-quotes-or-trailing-colon + value-after-colon.

**Example:**
```
"created:": 2024-03-14
```
Match groups: `("", "created", " 2024-03-14")`.

**Verdict contribution:** if any line matches → `BROKEN_KEYS_INSIDE_COLON`.

### Pattern 1b — Standard quoted-key (shape α — F25)

**Per-line regex (single-line input):**

```python
F25_STANDARD_QUOTED_PATTERN = re.compile(r'^(\s*)"([^":]+)"\s*:(.*)$')
```

The crucial part: `[^":]+` inside the brackets — NO `"` and NO `:` allowed inside the key name. This is what distinguishes shape α from shape β.

**Match:** indent + key-without-quotes-or-inside-colons + value-after-colon.

**Example:**
```
"description": A standard quoted-key value
```
Match groups: `("", "description", " A standard quoted-key value")`.

**Verdict contribution:** if any line matches AND no shape-β line matches → `OK_QUOTED`. If both shapes coexist on different lines → `BROKEN_KEYS_INSIDE_COLON` wins (the broken shape signals to the skill to repair, then re-scan).

**No repair needed.** Standard quoted-keys are valid YAML. Skill regexes must simply accept this shape (see § 5.1 detection-precedence rule and per-skill policy table).

### Pattern 2 — Multiple frontmatter blocks

Walk the entire file (not just frontmatter). Count occurrences of lines where
`.rstrip() == '---'`. Treat first occurrence at line 0 as frontmatter open.
Treat next occurrence as frontmatter close.

After the close, if a subsequent line is also `.rstrip() == '---'` AND is NOT
inside a code-fence (no `\`\`\`` opening before it), AND a subsequent matching
`---` exists: this is a SECOND frontmatter block.

**Verdict contribution:** if two or more frontmatter blocks → `MULTIPLE_FRONTMATTER_BLOCKS`.

This pattern is canonically defined in `note-rename`'s Corrupted File Detection
(SKILL.md § "Corrupted File Detection"). The sanity-check uses the same
detection logic, exposed as a callable.

### Pattern 3 — Unclosed frontmatter

If line 0 is `---` but no subsequent line equals `---`: frontmatter never
closes. **Verdict:** `UNCLOSED_FRONTMATTER`.

### Pattern 4 — Cosmetic-only (Class-D)

Whitespace-only lines between final field and closing `---`. Or trailing
whitespace on a frontmatter field line. These are non-blocking.

**Verdict contribution:** logged in finding list, but verdict stays at
the worst non-cosmetic verdict found.

## Idempotency

After a repair-skill runs § 5.1 inside-colon-quoted-key normalization, calling
the sanity-check again on the same file MUST return `OK`, `OK_QUOTED`, or
`OK_NO_FRONTMATTER` (any non-`BROKEN_KEYS_INSIDE_COLON` non-Class-A verdict).
This is the contract: repair is permanent within the run, normalize is idempotent.

## Recipe (f) — Normalize inside-colon quoted-keys (F26 repair)

This is the F26 cluster surface (shape β — inside-colon). Read it twice.

> **Scope:** Recipe (f) targets ONLY shape β (`"<key>:":` with inside-colon
> AND outside-colon). It does NOT touch shape α standard quoted-keys
> (`"<key>":` with no inside-colon — valid YAML). See `references/yaml-sanity.md`

```text
1. Read frontmatter lines (recipe a). If no frontmatter, no-op.
2. Walk frontmatter lines with index. For each line where the per-line regex
   F26_INSIDE_COLON_PATTERN = re.compile(r'^(\s*)"([^"]+):"\s*:(.*)$') matches:
   a. Extract groups: indent, key_name, value_with_colon_separator.
   b. Replace line with: f"{indent}{key_name}:{value_with_colon_separator}".
3. After all replacements, walk frontmatter lines again. If any key-name now
   appears on two or more lines (post-normalization collision):
   a. Keep the FIRST occurrence (= the line that was originally inside-colon-
      quoted, now normalized — assume original/correct value).
   b. Remove subsequent occurrences.
   c. Log each removed line as Class-D finding (file_ref + key_name + value).

   Re-running on already-normalized frontmatter returns unchanged.
5. Standard quoted-keys (shape α — `"description":` with no inside-colon) are
   untouched by this recipe. They are valid YAML and pass through.
6. Write frontmatter + body back per recipe (a) write-back semantics.
```

## DO NOT — broken normalize patterns

```python
# DO NOT WRITE THIS — multi-line regex over joined frontmatter
content = re.sub(
    r'(?ms)"([^"]+):"\s*:\s*(.+?)\n',
    lambda m: f'{m.group(1)}: {m.group(2)}\n',
    content,
)
# Why it fails: under (?s), `.+?` matches across newlines (non-greedy still
# matches newlines). The lazy quantifier saves you from greedy-eat-everything
# but YAML quoted-string values can themselves span newlines — corrupting
# multi-line values. Per-line regex is fine. Multi-line regex is not.
```

Use recipe (f) as defined. Walk lines. Per-line regex match. Replace per line.
```

## 7. `scripts/smoke-test.sh` Design

### 7.1 Goal

Detect regression of any v0.1.2 + v0.1.3 fix without manual GR-runs. Runs
locally on macOS (the dev environment). Cross-platform later (v0.2.0+).

### 7.2 Constraints

- Must NOT depend on actual Claude Code runtime — skills are markdown specs,
  not executable code. Smoke-test verifies the **input/output contract**, not
  the LLM execution path.
- Must not have external dependencies beyond standard macOS shell tools (bash,
  diff, mkdir, cp, rm).
- Exit 0 on full pass, exit 1 on first diff.

### 7.3 Architecture

The runner is a contract-validator, not an LLM-executor. It:

1. Sets up a temp vault with copies of `test-data/*.md`.
2. For each fixture, the runner expects a parallel `test-data/expected/<skill>/<fixture>.md` file documenting the post-skill state.
3. The runner does NOT actually invoke Claude Code or any LLM. Instead, it provides a manual workflow for the developer:
   - Print the temp-vault path
   - Print the next test-step (e.g. "Run `property-enrich` against `<temp-vault>` with `scope=vault`")
   - Wait for user to indicate done (`y` to continue)
   - Diff temp-vault state against `test-data/expected/<skill>/`
   - Report diff result

This is a "harness for manual GR-runs against fixtures" rather than full
automation. v0.2.0+ may add LLM-driven automation if Claude Agent SDK supports
it cleanly.

### 7.4 Script outline

```bash
#!/usr/bin/env bash
# scripts/smoke-test.sh
# Manual harness for v0.1.x test fixtures. Sets up temp vault, walks fixtures,
# diffs against expected golden files. Exits 1 on first diff.

set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
TEMP_VAULT="$(mktemp -d /tmp/vault-autopilot-smoke.XXXXXX)"
trap 'rm -rf "$TEMP_VAULT"' EXIT

echo "=== smoke-test ==="
echo "Repo root:  $REPO_ROOT"
echo "Temp vault: $TEMP_VAULT"
echo

# 1. Copy fixtures to temp vault
mkdir -p "$TEMP_VAULT/Inbox"
cp "$REPO_ROOT/test-data"/*.md "$TEMP_VAULT/Inbox/"
echo "Copied $(ls "$REPO_ROOT/test-data"/*.md | wc -l) fixtures to Inbox/"
echo

# 2. Walk skills in declared order
SKILLS=(property-enrich note-rename inbox-sort property-describe)

for SKILL in "${SKILLS[@]}"; do
  echo "--- Step: $SKILL ---"
  echo "Run \`$SKILL\` against \`$TEMP_VAULT\` (scope=inbox)."
  echo "Press Enter when done (or Ctrl-C to abort)..."
  read -r

  # 3. Diff against expected golden files
  EXPECTED_DIR="$REPO_ROOT/test-data/expected/$SKILL"
  if [[ ! -d "$EXPECTED_DIR" ]]; then
    echo "  No expected/ directory for $SKILL — skipping diff"
    continue
  fi

  for EXPECTED in "$EXPECTED_DIR"/*.md; do
    FIXTURE="$(basename "$EXPECTED")"
    ACTUAL="$TEMP_VAULT/Inbox/$FIXTURE"
    if [[ ! -f "$ACTUAL" ]]; then
      # File may have been moved by inbox-sort — search recursively
      ACTUAL="$(find "$TEMP_VAULT" -name "$FIXTURE" -type f | head -1)"
    fi
    if [[ ! -f "$ACTUAL" ]]; then
      echo "  FAIL: $FIXTURE not found in temp vault after $SKILL"
      exit 1
    fi
    if ! diff -u "$EXPECTED" "$ACTUAL" > /tmp/smoke-diff.txt; then
      echo "  FAIL: $FIXTURE differs from expected"
      cat /tmp/smoke-diff.txt
      exit 1
    fi
    echo "  PASS: $FIXTURE matches expected"
  done
done

echo
echo "=== smoke-test PASS ==="
```

### 7.5 Limitations

- Manual: developer runs each skill against the temp vault, then presses Enter. Real automation requires Claude Agent SDK integration (v0.2.0+).
- Skill-log callout includes timestamps — exact byte-match diff fails. Solution: golden files use placeholders (`{{TIMESTAMP}}`) and the diff filters those before comparing.
- Birthtime preservation tests need filesystem-level inspection (stat -f %B) — not in initial smoke-test.

These limitations are acceptable for v0.1.3. The smoke-test catches regression in the YAML-edit recipes (the F8/F15/F19/F25/F26 surface) — which is the bug-class v0.1.x is fighting.

## Appendix A — Living Decisions

Decisions made during spec-drafting that may want propagation to other docs:

1. **AD-X (TBD):** Quoted-key repair-vs-skip per skill is determined by the skill's
   own contract, not by a global policy. Repair-skills repair; additive-only
   skills skip-with-finding. The sanity-check is the inversion-of-control point.
2. **Cluster-handling philosophy:** "Defense in depth via per-line regex" — every
   skill that writes YAML calls sanity-check first. If sanity passes, recipes
   are safe. If sanity fails, skill knows what to do (repair or skip) per its
   contract. The cost is one extra read+walk per file; the benefit is no more
   F8/F15/F19/F25/F26-class bugs.
3. **Test-fixture-as-spec:** `test-data/<id>-repro.md` files are not just tests
   — they are the canonical reproductions. The expected/ golden files document
   the post-skill state precisely. Future bug discoveries → new fixture, new
   expected, regression-locked.
